# MDatos.ai database foundation

This folder contains the first PostgreSQL schema for the MDatos.ai operating platform.

## What this schema solves

- Charges the customer before GPU cost is incurred.
- Tracks prepaid credits with immutable ledger movements.
- Stores experiment history for a future "Mi Laboratorio" workspace.
- Records Vast.ai lease lifecycle and each compute job attempt.
- Keeps results in object storage while exposing only short-lived download grants.
- Preserves auditability for payments, worker callbacks, and user notifications.

## Core design choices

1. Multi-tenant by organization
Every operational table carries `organization_id`, so the backend can enforce tenant isolation and later add Row Level Security if needed.

2. Wallet plus immutable ledger
`credit_wallets` holds current balances for fast reads.
`credit_ledger` stores the source of truth for purchases, reservations, releases, charges, refunds, and manual adjustments.

3. Public tier, private orchestration profile
`service_plans` contains both the customer-facing package and the internal sourcing guardrails for Vast.ai:
- credit allowance
- queue priority
- runtime cap
- GPU, RAM, disk, and hourly cost constraints
- target margin ratio

4. Experiments separated from execution attempts
`experiments` is the customer project record.
`compute_jobs` captures retries and execution attempts.
`provider_leases` stores the actual rented GPU node lifecycle.

5. Secure artifact delivery
`result_artifacts` stores only metadata and object coordinates.
`download_grants` records each temporary access grant without exposing the storage bucket permanently.

## Workflow mapping

1. User creates experiment and uploads inputs.
`experiments` + `experiment_inputs`

2. User pays or consumes credits.
`payment_orders` + `payment_events` + `credit_wallets` + `credit_ledger`

3. Backend reserves credits before provisioning compute.
`credit_ledger` reserve entry + `compute_jobs`

4. Backend leases a Vast.ai machine and launches the container.
`provider_leases` + `compute_jobs`

5. Worker uploads final outputs to S3 or B2 and reports completion.
`result_artifacts` + `experiment_events`

6. Backend issues signed download access and notifies the user.
`download_grants` + `outbound_notifications`

## Important assumptions

- Postgres is the transactional source of truth.
- Authentication is handled by an external identity provider, not local passwords.
- Secrets for Vast.ai, Stripe, S3, or B2 should live in a secret manager, never in these tables.
- Upload and download URLs are generated by the backend at request time.
- The worker callback should be authenticated and idempotent.
